## Executive summary

The repo is in solid shape for a beta — the architecture is clean, the API surface is small, Sentry is wired up, and the family portal already enforces server-side IDOR protection. The main themes to address:

1. **Auth model is identifier-as-credential.** A raw user id is sent as `X-User-Id` from both mobile and web. The API code itself flags this as "v1 beta only." This should be replaced with a signed/opaque session token before any non-beta launch.
2. **`@techbuddy/shared` is scaffolded but unused.** Wire types are duplicated in `apps/mobile/lib/api.ts` and `apps/web/src/lib/api.ts` and are already drifting (different `IssueSummary` shapes, different `recommendRoute` enums). This is the single biggest source of latent bugs.
3. **Mobile carries Expo-template dead weight.** ~14 component/hook files plus 4–8 dependencies are unreferenced after onboarding the real app. ✅ **FIXED 2026-05-06** — boilerplate files deleted; orphan deps still pending.
4. **Web portal is "use client" everywhere with effect-driven fetches.** Once auth moves to a cookie, the dashboard and detail pages should become RSCs with `<Suspense>` for an instant first paint.
5. **No CI gate, no committed deploy config, no root `package.json`.** Render and Vercel auto-deploy from `main` with nothing checking typecheck/build first. ✅ **FIXED 2026-05-06** — `.github/workflows/ci.yml`, `render.yaml`, and `apps/web/vercel.json` all committed and CI is green. (Note: there was already a root `package.json`; the audit was off there.)

## P0 — security & prod-breaking

### API

**Auth identity = unsigned user id**
`apps/api/src/lib/auth.ts:42-69` — the `X-User-Id` header is the only credential and is a guessable cuid. The file's own docblock acknowledges this is beta-only. Combined with the CORS allowance below, anyone who learns or guesses an id impersonates that account.
→ Mint a signed session token (HMAC/JWT) using the already-declared `JWT_SECRET`; verify in the pre-handler; treat the raw header as a dev-only fallback.

**CORS allows requests with no `Origin` in production**
`apps/api/src/server.ts:38-49` — the `!origin` allowance is needed for native apps but should be tightened in production for browser-class requests (detect via `Sec-Fetch-Site` / `User-Agent`).
→ In prod, allow `!origin` only when no browser `User-Agent` is set.

**`/v1/debug/sentry-test` exposed in production**
`apps/api/src/routes/debug.ts:26-51` — gated only by the weak X-User-Id check; any user can spam Sentry events.
→ Gate by `NODE_ENV !== "production"` or require a `DEBUG_SECRET` query param (the file's own comment suggests this).

### Mobile

**User id stored in plain AsyncStorage and used as bearer**
`apps/mobile/lib/auth.tsx:14,48,73` + `apps/mobile/lib/api.ts:30-32` — anything that can read app storage steals the account.
→ Move the id to `expo-secure-store` (Keychain/Keystore) and replace it with a server-issued opaque token once auth is upgraded.

**Sentry `sendDefaultPii: true` plus diagnostic events include user id and name** ✅ **FIXED 2026-05-06**
`apps/mobile/app/_layout.tsx:17-29` and `apps/mobile/app/settings.tsx:215-223` — IP, cookies, and user info ship to Sentry; the diagnostic button explicitly attaches `user.id`/`user.name`. Bug-report breadcrumbs also include free-text the senior typed.
→ ✅ Set `sendDefaultPii: false` in `_layout.tsx`. Diagnostic button now logs only the last 6 chars of the user id, dropped `userName`. Added `lib/safe-error.ts` and converted all 21 `console.error("[X]", err)` call sites to `safeErrorMessage(err)` so breadcrumbs don't capture full error objects (which can echo back user input).

### Web

**Auth token in `localStorage` + no edge enforcement**
`apps/web/src/lib/auth-context.tsx:42-53` and `apps/web/src/lib/api.ts:16-30`; no `middleware.ts`. `dashboard/page.tsx:32-36` and `seniors/[id]/page.tsx:48-52` redirect after hydration — pages render to HTML before any auth check, and the credential lives in JS-readable storage.
→ Move to an HttpOnly, Secure, SameSite=Lax cookie set by the API (or a Next route handler), then add a `middleware.ts` that 302s unauthenticated requests away from `/dashboard` and `/seniors/*`.

**No security headers** ✅ **FIXED 2026-05-06 (late)**
`apps/web/next.config.mjs` has no `headers()` block — missing CSP, `X-Frame-Options`, `Referrer-Policy`, HSTS, `X-Content-Type-Options`. Portal can be iframed; no defense-in-depth against XSS.
→ ✅ Added a `headers()` export with: CSP (allows self + `NEXT_PUBLIC_API_URL` origin + `*.ingest.sentry.io` for future Sentry web hookup), `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, HSTS (1 year, includeSubDomains, preload), `Permissions-Policy` denying camera/mic/geolocation. CSP includes `upgrade-insecure-requests`. Worth testing the deployed page with [securityheaders.com](https://securityheaders.com) — should grade A.

## P1 — real bugs & significant waste

### API

**Anthropic image inputs uncapped** ✅ **FIXED 2026-05-06**
`apps/api/src/lib/buddy.ts:4`, `apps/api/src/routes/chat.ts:218-224`, `apps/api/src/routes/bug-reports.ts` — only the global `bodyLimit` (10 MB) applies; no per-image MB cap, no MIME-vs-prefix check. Bug-reports stores the base64 inline in Postgres forever.
→ ✅ Added `.max(7_000_000)` on the base64 string in `ImageInputSchema` for both `chat.ts` and `bug-reports.ts`. MIME-vs-prefix check still TODO; re-encoding before persistence still TODO.

**Family invite codes use `Math.random()` over 1M numeric keyspace** ✅ **FIXED 2026-05-06**
`apps/api/src/routes/family.ts:36-38` plus no rate limiting on `POST /v1/family/accept`. Brute-force claims a senior's pending invite.
→ ✅ Switched to `node:crypto`'s `randomInt`. Added per-IP rate limit (5/minute, 20/hour) on `/v1/family/accept` AND `POST /v1/users` via new `checkInviteAcceptRateLimit` / `checkUserCreateRateLimit` in `lib/rate-limit.ts`. Also enabled Fastify `trustProxy: true` so `request.ip` reflects the real caller behind Render's proxy. Brute-forcing the 6-digit keyspace from one IP now takes ~2,000+ hours instead of ~30 minutes. Widening to 8 alphanumeric chars deferred (not necessary with rate limiting).

**`sweepAbandoned` runs on every Home-screen load**
`apps/api/src/routes/sessions.ts:19-33` called from `:48` — every list request fires a write transaction with a correlated NOT EXISTS subquery before the read.
→ Move to a periodic job, or gate behind a "last swept" timestamp; add `@@index([status, startedAt])` on `Session` to make the sweep cheap.

**`userContext.findMany` per chat turn**
`apps/api/src/routes/chat.ts:163-166` — extra round-trip on the hot path for data that changes rarely.
→ Cache per-user in memory with a TTL; invalidate on PATCH/POST/DELETE in `user-context.ts`.

**Missing index for the `Session` sweep**
`prisma/schema.prisma:197-208` — current `[userId, startedAt]` doesn't support the abandoned-session full-table scan.
→ Add `@@index([status, startedAt])` (covers both the hot list and the sweep).

**No graceful shutdown / `unhandledRejection` handler**
`apps/api/src/server.ts:75-82` — SIGTERM from Render mid-request drops in-flight Anthropic calls and leaves the Prisma pool dangling.
→ `process.on('SIGTERM'/'SIGINT', () => fastify.close())` plus an `unhandledRejection` logger that funnels to Sentry.
