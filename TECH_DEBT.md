## Active shortcuts

### 1. Sentry source-map upload is disabled on EAS builds

**Where:** `apps/mobile/eas.json` — `SENTRY_DISABLE_AUTO_UPLOAD=true` in all three build profiles.

**Why:** `@sentry/cli`'s postinstall step fails on EAS Linux runners. The CLI binary either doesn't get downloaded or doesn't have the right permissions. We tried adding `pnpm.onlyBuiltDependencies` to the root `package.json` to force the postinstall to run; that fixed it locally on Windows but the Linux variant of `@sentry/cli` (`@sentry/cli-linux-x64`) still wasn't being installed on EAS. Suspected root cause: `pnpm-lock.yaml` only locks the platform variants present at the machine that generated the lockfile (Windows in our case).

**Impact:** Crashes still report to Sentry — the runtime SDK is independent of the CLI. But stack traces in the Sentry UI will be minified (e.g. `a.b.c` instead of `handleSubmit`), so debugging production crashes is harder.

**Fix path (when we have time):**
- **Option A (preferred):** Generate `pnpm-lock.yaml` on Linux to include the Linux `@sentry/cli` variant. Easiest way: run `pnpm install` inside WSL or in a CI container, commit the resulting lockfile.
- **Option B:** Manual source-map upload after each build via `npx sentry-expo-upload-sourcemaps` once the build artifact is downloaded.
- **Option C:** Lock `@sentry/cli` to a specific version known to work, or pre-download the Linux binary into the repo.

**Acceptance:** Remove the env var from all three profiles in `eas.json` and confirm the build completes with the `:app:createBundleReleaseJsAndAssets_SentryUpload_*` task running (not SKIPPED).

## Deprecations to watch

These are warnings emitted by libraries inside `node_modules/`. They're not ours to fix — they get resolved when we bump Expo SDK and the underlying packages release new versions. The list exists so we know what to verify after each major dependency upgrade.

### React Native New Architecture transition
- `ReactNativeHost` is deprecated in favor of `ReactHost` across `expo`, `expo-modules-core`, `react-native-screens`, and our autogenerated `MainApplication.kt`.
- Will resolve when Expo and React Native finish their New Architecture migration upstream. Likely SDK 55 or 56.

### Gradle 9 incompatibility
- The build emits "Deprecated Gradle features were used in this build, making it incompatible with Gradle 9.0." All deprecated calls are in expo and react-native gradle plugins, not ours.
- Will resolve as those plugins migrate. Not urgent — Gradle 9 isn't out yet.

### Android SDK 35+ status bar / nav bar APIs deprecated
- `react-native-screens` emits ~30 warnings about `setStatusBarColor`, `setNavigationBarColor`, etc. being no-ops on Android 15+.
- We target SDK 36, so these are already no-ops in our build. The warnings will go away when react-native-screens removes the deprecated paths.

### Kotlin DSL deprecations (`kotlinOptions` → `compilerOptions`)
- Inside `expo-modules-autolinking` and `expo-modules-core` gradle plugins.
- Resolves when Expo updates those internal plugins.

### `package=` attribute in AndroidManifest.xml
- `@react-native-async-storage/async-storage`, `react-native-safe-area-context`, and `@sentry/react-native` still set the `package` attribute that AGP no longer honors.
- Their problem, not ours. Resolves on their next release.

## Redundancies to clean up

### `apps/mobile/.env.local` has a `SENTRY_AUTH_TOKEN` we no longer need locally
The Sentry wizard wrote this file when we ran setup. It's now duplicated as an EAS project secret (`eas secret:list`), and EAS uses the secret, not the local file. The local copy isn't hurting anything (it's gitignored), but if we ever rotate the Sentry token, we need to update *both* places. Fix: delete the local file once you confirm `eas build` still succeeds without it (which it should — the secret is the source of truth).

## Future-proofing reminders

- **Bump Expo SDK on a regular cadence.** Each SDK bump is also when you should re-test the Sentry source-map upload, because new `@sentry/react-native` versions sometimes ship a fixed `@sentry/cli`.
- **Re-generate `pnpm-lock.yaml` on Linux periodically** (or move CI to Linux) to keep cross-platform optional dependencies locked.
- **Don't hand-edit `android/` or `ios/` folders.** They're regenerated by `expo prebuild`. If a deprecation warning points at `MainApplication.kt` or similar, that's Expo's responsibility, not yours.
- **Keep `node` pinned** in `eas.json` for every profile (currently `22.11.0`). Reproducible builds matter.
- **EAS secrets are the source of truth** for build-time tokens. `.env.local` is for local dev only.
