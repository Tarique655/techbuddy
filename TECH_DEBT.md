## Active shortcuts

### 1. Render container start is fragile against Neon cold starts

**Where:** `apps/api/package.json` — `"start": "prisma migrate deploy && node dist/server.js"`.

**Why:** Neon free-tier databases auto-suspend after ~5 min of inactivity. When the Render container restarts (which happens on every deploy and periodically on the free tier), `prisma migrate deploy` hits a cold Neon and sometimes fails with `P1001: Can't reach database server`. Manually re-deploying usually works on the second try because the DB is then warm — but that's a "babysit the deploy" workflow we shouldn't accept long-term.

**Impact:** Every deploy has a non-zero chance of failing for transient reasons unrelated to the code. Container restarts during low-traffic periods could leave the API down for minutes.

**Fix path:** Move migrations out of the start command into Render's **Pre-Deploy Command** (Render dashboard → Settings → Build & Deploy → Pre-Deploy Command). Set it to `pnpm --filter @techbuddy/api exec prisma migrate deploy`. Change start to just `node dist/server.js`. Pre-Deploy runs after build, before traffic shifts; if it fails, the new version doesn't go live and the old one keeps serving. Container restarts no longer touch the schema layer at all.

**Acceptance:** After making the change, deploy once, observe Pre-Deploy logs show migrate running there. Force a Render restart (manual restart button), confirm container comes up without re-running migrate.

### 2. Sentry source-map upload is disabled on EAS builds

**Where:** `apps/mobile/eas.json` — `SENTRY_DISABLE_AUTO_UPLOAD=true` in all three build profiles.

**Why:** `@sentry/cli`'s execution fails on EAS Linux runners. The error in the build log is:

```
Execution failed for task ':app:createBundleReleaseJsAndAssets_SentryUpload_*'.
> A problem occurred starting process 'command '.../node_modules/@sentry/cli/bin/sentry-cli''
```

We've tried two fix paths and both failed:

1. **Regenerated `pnpm-lock.yaml` on Linux** (via GitHub Codespaces, 2026-05-05) so the Linux `@sentry/cli-*` optional deps are properly locked. Lockfile now contains `@sentry/cli-linux-x64@3.4.1` and friends — verified via `Select-String`. Build still failed identically.
2. **Added platform-specific Sentry CLI packages to `pnpm.onlyBuiltDependencies`** in root `package.json` (`@sentry/cli-linux-x64`, `@sentry/cli-linux-arm64`, etc.) so their install scripts run. Build still failed identically.

The binary apparently exists at the expected path but Gradle can't start the process. Without log access to the EAS build environment we can't tell whether it's a permissions issue, a missing libc dependency, or something else in the @sentry/cli wrapper.

**Impact:** Crashes still report to Sentry — the runtime SDK is independent of the CLI. But stack traces in the Sentry UI will be minified (e.g. `a.b.c` instead of `handleSubmit`), so debugging production crashes is harder.

**Fix path (when we have time):**
- **Option A:** Manual source-map upload from Windows after each build. The local Windows `@sentry/cli` works fine (`pnpm exec sentry-cli sourcemaps upload ...`). Wire into a post-build script. Reliable but adds a manual step.
- **Option B:** Bump `@sentry/react-native` and `@sentry/cli` to the latest versions on the next Expo SDK bump. Maybe a newer release fixes whatever's wrong with the CLI dispatch on Linux.
- **Option C:** Switch the EAS build image to one that pre-installs sentry-cli system-wide, bypassing the npm-shipped binary entirely.
- **Option D:** Open an issue against `@sentry/react-native` with the exact error and our package.json + lockfile excerpt; may be a known issue with a workaround we haven't found.

**Acceptance:** Remove the env var from all three profiles in `eas.json`, push, run a fresh `eas build`, confirm the `:app:createBundleReleaseJsAndAssets_SentryUpload_*` task completes successfully (not failing on "A problem occurred starting process").

**Things we already tried that did NOT work** — don't re-attempt:
- Regenerating lockfile on Linux ✗
- Adding `@sentry/cli-linux-*` to `pnpm.onlyBuiltDependencies` ✗

## Deprecations to watch

These are warnings emitted by libraries inside `node_modules/`. They're not ours to fix — they get resolved when we bump Expo SDK and the underlying packages release new versions. The list exists so we know what to verify after each major dependency upgrade.

### React Native New Architecture transition
- `ReactNativeHost` is deprecated in favor of `ReactHost` across `expo`, `expo-modules-core`, `react-native-screens`, and our autogenerated `MainApplication.kt`.
- Will resolve when Expo and React Native finish their New Architecture migration upstream. Likely SDK 55 or 56.

### Gradle 9 incompatibility
- The build emits "Deprecated Gradle features were used in this build, making it incompatible with Gradle 9.0." All deprecated calls are in expo and react-native gradle plugins, not ours.
- Will resolve as those plugins migrate. Not urgent — Gradle 9 isn't out yet.

### Android SDK 35+ status bar / nav bar APIs deprecated
- `react-native-screens` emits ~30 warnings about `setStatusBarColor`, `setNavigationBarColor`, etc. being no-ops on Android 15+.
- We target SDK 36, so these are already no-ops in our build. The warnings will go away when react-native-screens removes the deprecated paths.

### Kotlin DSL deprecations (`kotlinOptions` → `compilerOptions`)
- Inside `expo-modules-autolinking` and `expo-modules-core` gradle plugins.
- Resolves when Expo updates those internal plugins.

### `package=` attribute in AndroidManifest.xml
- `@react-native-async-storage/async-storage`, `react-native-safe-area-context`, and `@sentry/react-native` still set the `package` attribute that AGP no longer honors.
- Their problem, not ours. Resolves on their next release.

## Redundancies to clean up

### `apps/mobile/.env.local` has a `SENTRY_AUTH_TOKEN` we no longer need locally
The Sentry wizard wrote this file when we ran setup. It's now duplicated as an EAS project secret (`eas secret:list`), and EAS uses the secret, not the local file. The local copy isn't hurting anything (it's gitignored), but if we ever rotate the Sentry token, we need to update *both* places. Fix: delete the local file once you confirm `eas build` still succeeds without it (which it should — the secret is the source of truth).

## Future-proofing reminders

- **Bump Expo SDK on a regular cadence.** Each SDK bump is also when you should re-test the Sentry source-map upload, because new `@sentry/react-native` versions sometimes ship a fixed `@sentry/cli`.
- **Re-generate `pnpm-lock.yaml` on Linux periodically** (or move CI to Linux) to keep cross-platform optional dependencies locked.
- **Don't hand-edit `android/` or `ios/` folders.** They're regenerated by `expo prebuild`. If a deprecation warning points at `MainApplication.kt` or similar, that's Expo's responsibility, not yours.
- **Keep `node` pinned** in `eas.json` for every profile (currently `22.11.0`). Reproducible builds matter.
- **EAS secrets are the source of truth** for build-time tokens. `.env.local` is for local dev only.

## Operational gotchas (lessons learned, not active debt)

These have already bitten us at least once. Documented so we recognize them faster next time.

- **Sentry SDK reporting `flushed: true` is not proof Sentry stored the event.** Sentry's ingest endpoint accepts requests for any DSN with a valid format, then silently drops them if the project id doesn't exist. If events go missing, FIRST verify the DSN's project id matches a real project in `teekaylabs.sentry.io/settings/projects/` — don't assume the SDK or transport is broken.
- **Render env-var changes don't always trigger a fresh deploy from the latest commit.** They sometimes just restart the running container with the new env. If a code change AND env change both need to take effect together, do a Manual Deploy after editing the env var.
- **Sentry Fastify auto-instrumentation needs Node's `--import` flag, not a top-level import.** ESM hoisting means a `import "./instrument.js"` in `server.ts` runs *after* Fastify has already loaded, missing Sentry's monkey-patch window. The start command must be `node --import ./dist/instrument.js dist/server.js`. The smoking gun is a `[Sentry] fastify is not instrumented` warning in startup logs.
- **Anthropic SDK requires its Node shims registered before any other SDK import.** When Sentry's `--import` is active, it can pull `@anthropic-ai/sdk` internals out of order. Fix is to put `import "@anthropic-ai/sdk/shims/node";` at the very top of `instrument.ts`, before Sentry init. Symptom is `Error: you must import '@anthropic-ai/sdk/shims/node' before importing anything else from @anthropic-ai/sdk` at startup.
